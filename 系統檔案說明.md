# 醫療病歷 RAG 系統 — 檔案說明文件

> 本文件說明每個檔案的功能、用到的技術與套件，方便對外說明或日後回顧。

---

## 一、系統概覽

這是一套**全離線**的醫療病歷 RAG（Retrieval-Augmented Generation）系統。
使用者在網頁輸入症狀描述，系統從歷史病例資料庫中找出最相似的案例，再由 LLM 根據這些案例生成診斷參考。

整體流程：

```
使用者輸入症狀
    ↓
PHI 掃描（去除個資）
    ↓
Embedding（轉換成向量）
    ↓
FAISS 向量搜尋（找出候選病例，top-20）
    ↓
Reranker 重排序（精選 top-5）
    ↓
Llama-3.2 生成回答
    ↓
Hallucination 信心評估
    ↓
PHI 最後掃描 → 回傳給前端
```

### `src/deidentifier.py`

**功能：PHI（個人識別資訊）去識別化**

這個模組的任務是把病歷或使用者輸入中的個資偵測出來，用佔位符取代。
是整個系統的「隱私防護層」，分三個時間點運作：
1. 建索引前：對原始病歷資料做完整去識別化
2. 查詢時：掃描使用者輸入，避免個資進入向量搜尋
3. 輸出後：掃描 LLM 生成的文字，確保不會把個資從 context 重組出來

**偵測的 PHI 類型：**
- 台灣身分證字號（`[A-Z][12]\d{8}`，例如 A123456789）
- 手機號碼（`09XX-XXX-XXX`）
- 市話（`0X-XXXX-XXXX`）
- 生日（民國/西元年月日）
- 台灣地址（含市/縣＋區/鄉/鎮＋路/街＋號）
- 緊急聯絡人姓名（關鍵字後面的中文名）
- 病患姓名（「病患：」「患者：」等關鍵字後面的中文名）

**替換規則：**
- 同一筆病歷內，同一個名字換成同一個佔位符（A→`[姓名_A]`、B→`[姓名_B]`），保持一致性
- 查詢/輸出掃描時，不需要一致性，直接換成 `[PATIENT_NAME]`

**三個對外方法：**

| 方法 | 輸入 | 輸出 | 使用時機 |
|------|------|------|----------|
| `deidentify_record(record)` | 原始病歷 dict | 去識別化後的 dict | 建索引前（靜態防護） |
| `scan_query(query)` | 使用者輸入字串 | (cleaned_query, detected_types) | 每次查詢進來時 |
| `scan_output(text)` | LLM 輸出字串 | (text, has_phi_warning) | 生成完畢後 |

**用到的套件：**
- `re`（Python 標準庫）：所有 PHI 偵測都用正規表示式實作，沒有外部依賴

### `src/chunker.py`

**功能：醫療文本章節切割（Section-based Chunking）**

這個模組把一筆完整的病歷（JSON 格式）切成兩種 chunk，是「Small-to-Big Retrieval」設計的核心。

**為什麼不用固定字數切割：**
醫療文本的語意是連貫的，「主訴」和「診斷」不能分開，「診斷」和「治療計畫」不能分開。
如果用固定字數切，會切到一半的句子，讓向量語意變差。

**兩種 chunk：**

| chunk_type | 包含欄位 | 用途 |
|------------|----------|------|
| `query` | 主訴 + 現病史 + 診斷 | 放進 FAISS 做向量搜尋（小，語意集中） |
| `context` | 診斷 + 治療計畫 + 用藥 + 結果 + 追蹤 | 取出來給 LLM 讀（大，資訊完整） |

**Small-to-Big 邏輯：**
用「小的 query chunk」搜尋，找到了哪筆病歷，再取那筆病歷的「大的 context chunk」給 LLM。
這樣搜尋精準，生成的資訊又完整。

**每個 chunk 帶的 metadata：**
```json
{
  "record_id": "...",
  "department": "內科",
  "visit_year": "2023",   ← 只保留年份，不留月日，避免重識別
  "chunk_type": "query"
}
```

**用到的套件：**
- 無外部依賴（純 Python 邏輯）

### `src/embedder.py`

**功能：將文字轉成向量（Embedding）**

把文字轉換成電腦看得懂的「數字向量」，相似的病例在向量空間中距離會很近。

**模型：**
- `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2`
- 支援 50+ 語言，包含繁體中文與中英混合的醫療術語
- 輸出維度：384（每段文字轉成 384 個數字）
- 使用 `normalize_embeddings=True`（歸一化），讓後面 FAISS 用內積就等於 cosine similarity

**兩個方法：**

| 方法 | 輸入 | 輸出 | 使用時機 |
|------|------|------|----------|
| `embed_texts(texts, batch_size=32)` | 文字 list | shape (N, 384) 的 numpy array | 建索引時，批次處理所有 query chunks |
| `embed_query(query)` | 單一字串 | shape (384,) 的 numpy array | 每次查詢時，把使用者輸入轉成向量 |

**用到的套件：**
- `sentence-transformers`：載入並執行 embedding 模型
- `numpy`：處理向量矩陣

### `src/retriever.py`

**功能：FAISS 向量資料庫建立與查詢**

管理 FAISS 向量索引，負責「把向量存進去」和「查詢最相似的病例」。

**FAISS 索引類型：**
- `IndexFlatIP`（Inner Product，內積）
- 因為向量已歸一化，內積就等於 cosine similarity（越高越相似，最大值為 1）
- 這是最精準的索引，不做近似，直接暴力比對所有向量

**儲存的兩個檔案：**
- `data/faiss_index/index.faiss`：向量索引本體
- `data/faiss_index/metadata.pkl`：query chunks 的文字內容 + context_map（record_id → context chunk）

**Small-to-Big 在這裡的實作：**
- `_query_chunks`：和 FAISS 索引一一對應，存著每個向量對應的 query chunk 文字
- `_context_map`：以 record_id 為 key，存著每筆病歷的 context chunk
- 搜尋時找到 query chunk，再透過 record_id 去 context_map 取對應的完整 context

**主要方法：**

| 方法 | 功能 |
|------|------|
| `build_index(chunks, embeddings)` | 建立索引並儲存到磁碟 |
| `register_context_chunks(chunks)` | 登記 context chunks（建索引時使用） |
| `load_index(index_path, metadata_path)` | 從磁碟載入索引（啟動服務時使用） |
| `search(query_embedding, top_k)` | 查詢最相似的 top-k 病例 |

**`search()` 回傳格式（每筆）：**
```json
{
  "chunk_id": "R001_query",
  "record_id": "R001",
  "content": "主訴：胸痛...",   ← query chunk 文字
  "context": "診斷：...",       ← 對應的 context chunk 文字（給 LLM 用）
  "score": 0.82,                ← cosine similarity
  "metadata": {...}
}
```

**用到的套件：**
- `faiss`（faiss-cpu）：向量索引核心
- `numpy`：向量運算
- `pickle`（Python 標準庫）：儲存/載入 metadata

### `src/reranker.py`

**功能：Cross-Encoder 重排序（第二層篩選）**

FAISS 是用向量距離找相似病例，速度快但不夠精準。
Reranker 對 FAISS 找出的候選病例逐一「深度評估」，重新排序取最相關的。

**兩階段設計（為什麼這樣做）：**
- 第一階段 FAISS：速度快（O(1)），先從幾千筆中找出候選（top-20）
- 第二階段 Reranker：精準但慢（O(N) 次 forward pass），只對候選做深度排序

**Reranker vs Embedding 的差異：**
- Embedding：query 和 document 分別轉成向量再比較（雙塔架構）
- Cross-Encoder：把 (query, document) 一起餵進模型，讓模型直接判斷相關性（更精準）

**模型：**
- `cross-encoder/mmarco-mMiniLMv2-L12-H384-v1`
- 多語言，支援繁體中文＋醫學英文
- 用 `content`（症狀+診斷摘要）做排序，不用 `context`（治療計畫）

**`rerank()` 方法：**
- 輸入：query 字串 + FAISS 候選 list
- 輸出：重排後的 top-k list，每筆多一個 `rerank_score` 欄位，原始 `score`（FAISS cosine）保留

**用到的套件：**
- `sentence-transformers`（`CrossEncoder` 類別）：cross-encoder 模型載入與推理

### `src/generator.py`

**功能：Llama-3.2-3B-Instruct 文字生成**

把 Retriever + Reranker 找到的病例 context 組成 prompt，讓 Llama 根據歷史病例生成診斷參考。

**模型載入（三種情況自動選擇）：**

| 環境 | 載入方式 | 記憶體需求 | 速度 |
|------|----------|------------|------|
| 有 GPU + use_4bit=True | 4-bit 量化（NF4）+ float16 計算 | 約 2.5GB VRAM | 快 |
| 有 GPU + use_4bit=False | float16 | 約 6GB VRAM | 快 |
| 無 GPU（CPU） | float32 | 約 12GB RAM | 很慢（每次約 200–400 秒） |

**System Prompt 設計要點：**
- 明確告知 LLM 這是醫療資訊輔助系統
- 規定只能根據提供的病例回答，不得憑空捏造
- 禁止輸出個人識別資訊
- 回答要標注資訊來源（「根據相似病例顯示...」）
- 涉及用藥劑量，必須提示需醫師確認

**`generate()` 流程：**
1. 把多個 context chunks 組成「相似病例資料」文字區塊
2. 組合成 messages（system + user）
3. 用 `apply_chat_template()` 格式化成 Llama 的 Instruct 格式
4. 傳入 `attention_mask`（Llama tokenizer 的 pad_token 和 eos_token 相同，不傳會有警告）
5. 用 `torch.no_grad()` 推理，輸出去掉 input tokens 後的新 token 解碼

**用到的套件：**
- `transformers`：`AutoModelForCausalLM`、`AutoTokenizer`、`BitsAndBytesConfig`（4-bit 量化）
- `torch`：GPU 偵測、推理、記憶體管理
- `os`（標準庫）：讀取 `HF_TOKEN` 環境變數

### `src/hallucination_checker.py`

**功能：LLM 回答的幻覺偵測與信心評分**

LLM 有時會「一本正經地說錯話」。這個模組評估 LLM 的回答有多少依據，給出信心等級。

**評估維度一：Retrieval Score（來自 FAISS）**
- 取所有相似病例中最高的 cosine similarity
- 分數高 = 資料庫有相似案例；分數低 = 可能是胡亂推斷

**評估維度二：Faithfulness Score（文字重疊）**
- 計算「回答中提到的醫療詞彙」有多少比例也出現在 retrieved context 中
- 公式：`出現在 context 的醫療詞彙數 / 回答中所有醫療詞彙數`
- 比例高 = 回答有所依據；比例低 = 可能在捏造

**醫療詞彙清單（內建 hardcode）：**
疾病（糖尿病、高血壓、心肌梗塞...）、症狀（發燒、咳嗽、胸痛...）、
藥物（aspirin、metformin、胰島素...）、檢查（血糖、CT、HbA1c...）、處置（手術、化療、透析...）

**信心等級判斷邏輯：**

| 等級 | 條件 | 前端顯示 |
|------|------|----------|
| HIGH | retrieval ≥ 0.75 AND faithfulness ≥ 0.6 AND 無劑量/強烈建議詞 | 綠色 |
| MEDIUM | retrieval ≥ 0.6 OR faithfulness ≥ 0.5，且無劑量/強烈建議詞 | 黃色 |
| LOW | 其他所有情況 | 橙色（建議人工確認） |

**觸發 LOW 的額外條件：**
- 回答包含具體劑量數字（如 `5mg`、`100ml`）
- 回答包含強烈建議語氣（「必須」「建議立即」「緊急」「馬上」）

**用到的套件：**
- `re`（標準庫）：偵測劑量數字與強烈建議詞的 pattern

### `src/pipeline.py`

**功能：整合所有模組的主要流程控制**

這是系統的「大腦」，把所有模組串起來，控制每次查詢的完整流程。

**初始化時載入的模組：**
- `MedicalDeidentifier`
- `MedicalEmbedder`
- `MedicalRetriever`（同時載入 FAISS 索引）
- `MedicalReranker`
- `MedicalGenerator`（Llama 在這裡載入，最耗記憶體）
- `HallucinationChecker`

**`query()` 完整流程（共 8 步）：**

```
1. scan_query()         → 偵測使用者輸入中的 PHI，自動遮蔽
2. embed_query()        → 把查詢文字轉成向量
3. retriever.search()   → FAISS 搜尋 top-20 候選病例
4. 低相似度檢查         → 若 top-1 的 cosine < 0.5，直接回傳「無相似病例」，不浪費 LLM 時間
5. reranker.rerank()    → 重排候選，縮減到 top-5
6. generator.generate() → Llama 根據 top-5 的 context 生成回答
7. checker.check()      → 評估信心等級，產生警示訊息
8. scan_output()        → 掃描 LLM 輸出，確保無個資洩漏
```

**`query()` 回傳格式：**
```python
{
  "answer": str,               # LLM 生成的回答（已過 PHI 掃描）
  "retrieved_cases": list,     # 相似病例摘要（含 summary + context）
  "confidence": str,           # HIGH / MEDIUM / LOW
  "retrieval_score": float,    # 最高 cosine similarity
  "warnings": list[str],       # 所有警示訊息合併
  "requires_review": bool,     # 是否需要人工確認
  "phi_detected_in_query": bool # 使用者輸入是否含個資
}
```

**Ollama 支援（備用模式）：**
- 若環境變數 `USE_OLLAMA=1`，改用 `src/generator_ollama.py`（Ollama API 版本）
- 其他模組完全不變，只換掉 generator

**用到的套件：**
- `os`（標準庫）：讀取 `USE_OLLAMA` 環境變數
- `torch`：偵測 GPU，決定 reranker/generator 的裝置

### `scripts/build_index.py`

**功能：一次性建立向量索引（離線前置作業）**

這是在啟動 Web 服務之前必須執行的腳本。只需執行一次（病歷資料更新時重跑）。

**執行指令：**
```bash
python scripts/build_index.py
python scripts/build_index.py --records data/records.json --output data/faiss_index
```

**6 個步驟：**
1. 載入 `data/records.json`（原始病歷）
2. 對每筆病歷執行 `deidentify_record()`（去識別化）
3. 對每筆去識別化病歷執行 `chunk_record()`（切割成 query + context chunk）
4. 對所有 query chunks 的文字執行 `embed_texts()`（批次轉換成向量）
5. 建立 FAISS 索引，同時把 context chunks 登記進 `context_map`
6. 印出統計資訊（病歷筆數、chunk 數量、向量維度、索引大小）

**輸出的檔案：**
- `data/faiss_index/index.faiss`
- `data/faiss_index/metadata.pkl`

**注意：context chunks 不進 FAISS**（只有 query chunks 進），這是 Small-to-Big 設計。
metadata.pkl 裡同時存著 context_map，供查詢時取用。

**用到的套件：**
- `argparse`（標準庫）：支援命令列參數
- `json`（標準庫）：讀取病歷 JSON
- `sys`（標準庫）：把 src 路徑加入 import 路徑

### `app.py`

**功能：FastAPI Web 後端**

提供 Web 服務介面，讓前端可以透過瀏覽器操作系統。

**三個 Route：**

| 路由 | 方法 | 功能 |
|------|------|------|
| `GET /` | GET | 渲染主查詢頁面 `index.html` |
| `POST /query` | POST | 接收查詢，呼叫 pipeline，寫 log，回傳 JSON |
| `GET /logs` | GET | 讀取查詢紀錄，渲染 `logs.html` |

**Lazy Loading 設計：**
- Pipeline（含 Llama 模型）不在啟動時就載入，而是在第一次 `/query` 時才初始化
- 原因：Llama 載入需要 10–30 秒，如果在啟動時就載入，會讓 `uvicorn` 的 startup 時間很長
- 如果索引檔案不存在，回傳 HTTP 503 + 說明訊息（不會噴 FAISS 的 C++ 錯誤）

**查詢 Log（`data/query_log.jsonl`）：**
- 每次查詢後寫入一行 JSON
- 記錄：時間戳、查詢內容、信心等級、最高相似度、警示數、是否需人工確認
- 注意：**不記錄 phi_detected**（隱私考量）

**`format_cases()` 輔助函式：**
- 把 pipeline 回傳的 retrieved_cases 轉成前端友善的格式
- 把 `score`（0~1 的 float）轉成百分比字串（例如 `"77.9%"`）
- 加上 department + visit_year 組合成 `meta` 顯示字串

**範例查詢（寫死在 app.py 裡）：**
4 個常見醫療情境，讓使用者可以快速測試系統。

**啟動指令：**
```bash
python app.py
# 或
uvicorn app:app --host 0.0.0.0 --port 7860
```

**用到的套件：**
- `fastapi`：Web 框架、路由、request/response 處理
- `uvicorn`：ASGI 伺服器（FastAPI 的執行環境）
- `jinja2`（透過 `fastapi.templating`）：渲染 HTML 模板
- `pydantic`：`QueryRequest` 資料驗證（`BaseModel`）
- `json`（標準庫）：讀寫 JSONL log
- `pathlib`（標準庫）：路徑操作

### `templates/index.html`

**功能：主查詢介面（純 HTML/CSS/JS，無框架）**

**版面：兩欄（1:2）**
- 左欄：症狀描述輸入框、查詢按鈕、範例查詢清單（點擊自動填入）
- 右欄：信心等級指示、LLM 回答、相似病例 accordion、警示訊息、免責聲明

**前端行為（純 JS，無 fetch 框架）：**
- `runQuery()`：POST 到 `/query`，把回傳結果動態更新右欄（不重整頁面）
- `Ctrl + Enter`（或 `Cmd + Enter`）：送出查詢
- 查詢中：按鈕 disabled + 顯示 spinner
- `renderCases()`：把相似病例組成 accordion 卡片，分「症狀與診斷」和「治療計畫與結果」兩區塊
- `renderWarnings()`：警示訊息用橙色橫幅顯示
- `setConfidence()`：根據 HIGH/MEDIUM/LOW 切換彩色圓點

**Markdown 渲染：**
- 使用 `marked.js`（CDN 載入）渲染 LLM 回答中的 Markdown
- LLM 的回答可能包含列表、粗體等 Markdown 格式

**安全注意：**
- `escHtml()` 函式：所有從後端取得的字串，在插入 HTML 前都做 escape，防止 XSS

**用到的技術/套件：**
- `marked.js`（CDN）：Markdown → HTML 渲染
- CSS Grid（`grid-template-columns: 1fr 2fr`）：兩欄版面
- Vanilla JS（原生 JavaScript，無 jQuery/React）

### `templates/logs.html`

**功能：查詢紀錄頁（Jinja2 Server-side Render）**

**上方統計卡片：**
- 總查詢次數
- HIGH / MEDIUM / LOW 信心各自次數（用 Jinja2 `selectattr` filter 計算）

**查詢紀錄表格欄位：**
- 時間、查詢內容、信心等級（badge）、最高相似度、警示數、是否需人工確認

**前端搜尋功能（純 JS）：**
- 搜尋框即時過濾表格列（不發 request 給後端）
- 顯示「顯示 X / 總共 Y 筆」的計數

**用到的技術：**
- Jinja2 模板語法：`{% for %}` 迴圈、`{% if %}` 判斷、`{{ }}` 變數輸出
- Jinja2 Filter：`selectattr`、`list`、`length`、`format`
- Vanilla JS

## 三、技術架構對應表

| 技術/套件 | 用在哪個模組 | 做什麼事 |
|----------|------------|---------|
| **正規表示式（re）** | deidentifier | 偵測台灣格式的 PHI |
| **SentenceTransformer** | embedder | 文字 → 384 維向量 |
| **CrossEncoder** | reranker | (query, doc) → 相關性分數 |
| **FAISS IndexFlatIP** | retriever | 向量相似度搜尋 |
| **pickle** | retriever | 儲存/載入 chunk metadata |
| **AutoModelForCausalLM** | generator | 載入 Llama-3.2-3B 模型 |
| **BitsAndBytesConfig** | generator | 4-bit NF4 量化（省 VRAM） |
| **FastAPI** | app | Web 路由、API 端點 |
| **Jinja2** | app + templates | HTML 模板渲染 |
| **marked.js** | index.html | LLM 回答的 Markdown 渲染 |
| **Vanilla JS Fetch** | index.html | 非同步送出查詢、動態更新畫面 |
